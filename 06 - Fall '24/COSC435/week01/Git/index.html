Basics

4 states

  • Untracked
  • Tracked
  • Staged
  • Commited

Untracked

  • Git doesn’t keep track of these.
  • All files are initially in this state.
  • Make git aware of these files using /add/

Tracked

  • Git is aware of these files.
  • Changes to these files will be noticed by git.

Staged

  • Files in the staged state will be commited in the next commit.
  • Use
git add

to move a file/directory into the staged state.

Commited

  • Once you make a commit, all the files in the staged state are included in the git history tree.
  • Each commit is given a SHA1 hash and a parent (the previous commit)
  • You can view this commit (the entire state of your project at this point in time) by checking out this commit!

History

  • Git history is a directed graph of commits
  • Every time you commit, a child (your commit) is added to the end of the graph/history
  • View this history with
git log
  • The history can take different paths (branches)

Basic Commands

init

  • Initializes a new git repository
  • Always add and commit a project before changing anything!
$ git init

status

  • Lists the status of your current repository
  • Shows which (tracked) files have changed and which files are staged for commit.
$ git status

add

  • Stages files/directories from your repo
  • Important: this only specifies which files will be commited in THE NEXT COMMIT!
$ git add .
$ git add somefile.java
$ git add somedir/

commit

  • Creates a new snapshot of your staged files. Assigns a hash. Adds this as a child to the current branch
$ git commit -m 'Your commit message'
$ git commit #(this will open vim for editing, watch out!)

log

  • View the commit history
$ git log
$ git log --oneline --decorate --all --graph

tag

  • Tagging assigns a name to a specific commit.
$ git tag <tag-name>
$ git tag lab1_complete

Remotes

  • Everything up to this point has occured locally
  • Remotes are servers that host your repo.
  • Very useful for accessing the same work on multiple PCs.
  • Also useful when working in teams.
  • git push and git pull are used to download (pull) and upload (push) your commits to the remote server.

push

  • Pushes a commit to a remote server
  • Everything is local until you push!
$ git push origin main
$ git push origin some_tag
$ git push origin some_other_branch
$ git push origin main --tags

pull

  • Pulls (downloads) any changes from your remote server (and merges those changes with your current branch)
  • Conflict can occur. Watch out!
$ git pull
$ git pull origin main

Project setup (git)

Create remote

  • First, create a new repository on Bitbucket.

Create a new project in Android Studio

  • The Android project structure will be created.
  • Navigate to the root of that project (it has the gradle.bat file in it)

Initialize your git repository

git init
  • (MAKE SURE YOUR AT THE ROOT OF YOUR PROJECT!)
  • This is usually done by Android Studio!

Add the remote

  • Now add your Bitbucket address as the remote of this git repository
  • git remote add origin http://bitbucket.org/…

Commit the initial files

  • ALWAYS DO THIS! This way, you can return to an initial state later
  • git add . (this adds everything in the current directory)
  • git commit -m ‘Some message’

Push your files to Bitbucket

  • git push origin master

Collaboration

Branches

  • Your git repo can take different paths through time
  • Create a new branch with
git checkout -b branch_name
  • You can then switch between branches and your working directory will reflect the different snapshots

Merging

  • Branches that have diverged can later be merged back together
  • Possible to have conflicts if the same file was changed in both branches being merged
